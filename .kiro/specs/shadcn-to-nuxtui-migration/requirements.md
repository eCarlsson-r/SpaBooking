# Requirements Document

## Introduction

This feature covers the full migration of the Nuxt 3 SPA/SSR application from shadcn/vue to NuxtUI. The project already has `@nuxt/ui` installed and an `app-nuxtui-layer/` directory with partial NuxtUI components (NavBar, Footer, etc.). The migration involves replacing all shadcn/vue primitive components (`button`, `card`, `dialog`, `drawer`, `input`, `label`, `skeleton`) and their usages across pages and components with their NuxtUI equivalents, removing shadcn-specific dependencies and configuration, and ensuring visual and functional parity throughout the application.

## Glossary

- **Migration_Tool**: The automated process (scripts or manual steps) that replaces shadcn/vue component references with NuxtUI equivalents across the codebase.
- **Shadcn_Component**: Any Vue component located in `app/components/ui/` that was generated by shadcn/vue (Button, Card, Dialog, Drawer, Input, Label, Skeleton and their sub-components).
- **NuxtUI_Component**: A component provided by the `@nuxt/ui` module, auto-imported and prefixed with `U` (e.g., `UButton`, `UCard`, `UModal`, `UDrawer`, `UInput`, `USkeleton`).
- **Consumer_Component**: Any `.vue` file outside `app/components/ui/` that imports or uses a Shadcn_Component.
- **App_Layer**: The `app/` directory containing pages, components, composables, stores, and assets.
- **NuxtUI_Layer**: The `app-nuxtui-layer/` directory containing NuxtUI-based nav and footer components.
- **Tailwind_Config**: The `tailwind.config.ts` file and `app/assets/css/tailwind.css` that define design tokens and global styles.
- **Shadcn_Config**: The `components.json` file and the `shadcn` block in `nuxt.config.ts` that configure the shadcn/vue integration.
- **CSS_Variable**: A CSS custom property (e.g., `--primary`, `--background`) used by shadcn/vue for theming.
- **Toast_Notification**: A transient UI notification currently triggered via `vue-sonner`'s `toast()` function in Consumer_Components.

## Requirements

### Requirement 1: Replace Shadcn Button with NuxtUI Button

**User Story:** As a developer, I want all Button usages to use `UButton` from NuxtUI, so that the codebase has a single, consistent button component.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all imports of `Button` from `@/components/ui/button` with the auto-imported `UButton` component.
2. WHEN a Consumer_Component previously used `<Button variant="..." size="..." class="...">`, THE Migration_Tool SHALL replace it with `<UButton variant="..." size="..." class="...">` using NuxtUI's equivalent prop names.
3. THE Migration_Tool SHALL delete the `app/components/ui/button/` directory and all files within it after all Consumer_Components have been updated.
4. WHEN the application renders a page containing a button, THE Application SHALL display a button that is visually consistent with the NuxtUI design system.

---

### Requirement 2: Replace Shadcn Card with NuxtUI Card

**User Story:** As a developer, I want all Card usages to use `UCard` from NuxtUI, so that card layouts are consistent and maintainable.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all imports of `Card`, `CardContent`, `CardHeader`, `CardTitle`, `CardDescription`, `CardFooter` from `@/components/ui/card` with the auto-imported `UCard` component.
2. WHEN a Consumer_Component previously composed a card using multiple sub-components (`CardHeader`, `CardContent`, etc.), THE Migration_Tool SHALL refactor the template to use `UCard`'s slot-based API (`#header`, `#default`, `#footer`).
3. THE Migration_Tool SHALL delete the `app/components/ui/card/` directory and all files within it after all Consumer_Components have been updated.
4. WHEN the application renders a card, THE Application SHALL display a card that is visually consistent with the NuxtUI design system.

---

### Requirement 3: Replace Shadcn Dialog with NuxtUI Modal

**User Story:** As a developer, I want the login modal and any other dialogs to use `UModal` from NuxtUI, so that modal behavior is consistent with the rest of the NuxtUI system.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all imports of `Dialog`, `DialogContent`, `DialogHeader`, `DialogTitle`, `DialogDescription`, `DialogFooter`, `DialogClose`, `DialogTrigger` from `@/components/ui/dialog` with the auto-imported `UModal` component.
2. WHEN a Consumer_Component previously used `<Dialog :open="..." @update:open="...">`, THE Migration_Tool SHALL replace it with `<UModal v-model="...">` binding the open state via a computed boolean ref.
3. THE Migration_Tool SHALL delete the `app/components/ui/dialog/` directory and all files within it after all Consumer_Components have been updated.
4. WHEN a user triggers the login modal, THE Application SHALL display a modal dialog that opens and closes correctly using NuxtUI's `UModal`.

---

### Requirement 4: Replace Shadcn Drawer with NuxtUI Slideover

**User Story:** As a developer, I want the booking drawer to use `USlideover` from NuxtUI, so that the slide-in panel behavior is consistent with the NuxtUI system.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all imports of `Drawer`, `DrawerContent`, `DrawerClose`, `DrawerHeader`, `DrawerTitle`, `DrawerDescription`, `DrawerFooter`, `DrawerOverlay` from `@/components/ui/drawer` with the auto-imported `USlideover` component.
2. WHEN a Consumer_Component previously used `<Drawer v-model:open="...">` with `<DrawerContent>`, THE Migration_Tool SHALL replace it with `<USlideover v-model="...">` with content placed in the default slot.
3. THE Migration_Tool SHALL delete the `app/components/ui/drawer/` directory and all files within it after all Consumer_Components have been updated.
4. WHEN a user initiates a booking, THE Application SHALL display a slide-over panel that opens from the bottom or side using NuxtUI's `USlideover`.

### Requirement 5: Replace Shadcn Input with NuxtUI Input

**User Story:** As a developer, I want all form inputs to use `UInput` from NuxtUI, so that form styling is consistent and the `@vueuse/core` `useVModel` dependency in the shadcn Input is removed.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all imports of `Input` from `@/components/ui/input` with the auto-imported `UInput` component.
2. WHEN a Consumer_Component previously used `<Input v-model="..." type="..." placeholder="..." class="...">`, THE Migration_Tool SHALL replace it with `<UInput v-model="..." type="..." placeholder="..." class="...">`.
3. THE Migration_Tool SHALL delete the `app/components/ui/input/` directory and all files within it after all Consumer_Components have been updated.
4. WHEN a user interacts with a form input, THE Application SHALL accept user input and update the bound model value correctly using NuxtUI's `UInput`.

### Requirement 6: Replace Shadcn Label with NuxtUI Label

**User Story:** As a developer, I want all form labels to use NuxtUI's label approach, so that the shadcn Label dependency is removed.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all imports of `Label` from `@/components/ui/label` with a native `<label>` HTML element or NuxtUI's form label pattern.
2. THE Migration_Tool SHALL delete the `app/components/ui/label/` directory and all files within it after all Consumer_Components have been updated.
3. WHEN a form is rendered, THE Application SHALL display labels that are visually and functionally equivalent to the previous shadcn Label component.

---

### Requirement 7: Replace Shadcn Skeleton with NuxtUI Skeleton

**User Story:** As a developer, I want all loading skeleton placeholders to use `USkeleton` from NuxtUI, so that loading states are consistent with the NuxtUI design system.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all imports of `Skeleton` from `@/components/ui/skeleton` with the auto-imported `USkeleton` component.
2. WHEN a Consumer_Component previously used `<Skeleton class="...">`, THE Migration_Tool SHALL replace it with `<USkeleton class="...">`.
3. THE Migration_Tool SHALL delete the `app/components/ui/skeleton/` directory and all files within it after all Consumer_Components have been updated.
4. WHEN data is loading, THE Application SHALL display animated skeleton placeholders using NuxtUI's `USkeleton`.

---

### Requirement 8: Replace vue-sonner Toast with NuxtUI Toast

**User Story:** As a developer, I want toast notifications to use NuxtUI's `useToast` composable, so that the `vue-sonner` dependency can be removed.

#### Acceptance Criteria

1. WHEN the Migration_Tool processes Consumer_Components, THE Migration_Tool SHALL replace all calls to `toast()` from `vue-sonner` with calls to `useToast().add()` from NuxtUI.
2. WHEN a Consumer_Component previously called `toast({ title, description, variant: 'destructive' })`, THE Migration_Tool SHALL replace it with `useToast().add({ title, description, color: 'red' })`.
3. THE Migration_Tool SHALL remove the `vue-sonner` import and dependency after all Consumer_Components have been updated.
4. WHEN a toast notification is triggered, THE Application SHALL display a toast notification using NuxtUI's toast system.

---

### Requirement 9: Remove Shadcn Configuration and Dependencies

**User Story:** As a developer, I want all shadcn/vue configuration and dependencies removed from the project, so that the codebase no longer has any shadcn/vue artifacts.

#### Acceptance Criteria

1. WHEN all Shadcn_Components have been replaced, THE Migration_Tool SHALL remove the `shadcn-nuxt` module from the `modules` array in `nuxt.config.ts`.
2. WHEN all Shadcn_Components have been replaced, THE Migration_Tool SHALL remove the `shadcn` configuration block from `nuxt.config.ts`.
3. THE Migration_Tool SHALL delete the `components.json` file from the project root.
4. THE Migration_Tool SHALL remove `shadcn-nuxt`, `class-variance-authority`, `vaul-vue`, and `vue-sonner` from `package.json` dependencies.
5. WHEN the project is built after migration, THE Application SHALL build successfully with no references to removed shadcn packages.

---

### Requirement 10: Migrate Shadcn CSS Variables to NuxtUI Theme

**User Story:** As a developer, I want the shadcn CSS custom properties removed from the global stylesheet and replaced with NuxtUI's theming system, so that there is no conflicting or redundant theming configuration.

#### Acceptance Criteria

1. WHEN the migration is complete, THE Migration_Tool SHALL remove all shadcn CSS variable declarations (`:root { --background: ...; --foreground: ...; ... }` and `.dark { ... }`) from `app/assets/css/tailwind.css`.
2. WHEN the migration is complete, THE Migration_Tool SHALL remove the `tailwindcss-animate` dependency if it is no longer required by any remaining component.
3. WHEN the application is rendered, THE Application SHALL apply consistent light and dark mode theming using NuxtUI's built-in color mode system.
4. WHEN the `colorMode` preference is set to `'light'` in `nuxt.config.ts`, THE Application SHALL render in light mode by default.

---

### Requirement 11: Remove Unused Utility and Library Files

**User Story:** As a developer, I want the `app/lib/utils.ts` file and other shadcn-specific utilities removed if they are no longer used, so that the codebase is clean.

#### Acceptance Criteria

1. WHEN all Shadcn_Components have been removed, THE Migration_Tool SHALL verify that `app/lib/utils.ts` (containing `cn()` and `valueUpdater()`) is no longer imported by any file.
2. IF `app/lib/utils.ts` has no remaining importers, THEN THE Migration_Tool SHALL delete the file and the `app/lib/` directory.
3. THE Migration_Tool SHALL remove `clsx`, `tailwind-merge`, and `radix-vue` from `package.json` dependencies if they are no longer referenced anywhere in the codebase.
4. WHEN the project is built after migration, THE Application SHALL build successfully with no unresolved import errors.

---

### Requirement 12: Preserve Existing NuxtUI Layer Components

**User Story:** As a developer, I want the existing `app-nuxtui-layer/` components to remain functional and unchanged during the migration, so that the NavBar, Footer, and navigation components continue to work correctly.

#### Acceptance Criteria

1. WHILE the migration is in progress, THE Migration_Tool SHALL not modify any files within the `app-nuxtui-layer/` directory.
2. WHEN the migration is complete, THE Application SHALL render the NavBar, Footer, and navigation components from `app-nuxtui-layer/` without errors.
3. WHEN a user navigates the application, THE Application SHALL display the `USlideover` mobile menu from `NavBar.vue` correctly.

---

### Requirement 13: Maintain Visual and Functional Parity

**User Story:** As a developer, I want the migrated application to look and behave the same as before the migration, so that end users experience no regressions.

#### Acceptance Criteria

1. WHEN the migration is complete, THE Application SHALL render all pages without JavaScript console errors related to missing or unresolved components.
2. WHEN a user interacts with the booking flow, THE Application SHALL open the booking slide-over, navigate between steps, and close the panel correctly.
3. WHEN a user submits the login form, THE Application SHALL open the login modal, accept credentials, and close the modal on success.
4. WHEN a user views the cart, THE Application SHALL display cart items with correct quantity controls and toast notifications on validation errors.
5. WHEN the application is built for production, THE Application SHALL produce a successful build with no TypeScript or module resolution errors.
